RULE 1 — BUILD INCREMENTALLY, PHASE BY PHASE
  You will execute this project across 9 defined phases (see Section 4).
  Complete one phase fully before starting the next. Do not skip ahead.
  Do not build Phase 3 code while Phase 2 is incomplete. Phases are gates.

RULE 2 — STOP AND ASK FOR THIRD-PARTY CONNECTIONS
  The moment any phase requires a third-party service credential, connection
  string, API key, or external account setup — STOP IMMEDIATELY. Do not
  attempt to mock it, skip it, or continue. Instead:

    (a) Tell the human exactly which service needs to be set up.
    (b) Provide a complete, numbered, step-by-step walkthrough for how
        the human should set it up — including where to click, what to
        copy, and what to name things.
    (c) Tell the human exactly what information to send back to you.
    (d) Wait. Do not proceed until you receive confirmation and the
        required values.

  Third-party services that will require this STOP protocol include:
    - Neon DB (PostgreSQL database setup and connection string)
    - TMDB API (account creation and API key)
    - RAWG API (account creation and API key)
    - Cloudinary (account creation, cloud name, API key, API secret)
    - Netlify (account and deployment configuration)
    - Render (account and deployment configuration)

RULE 3 — WRITE A PHASE REPORT AFTER EVERY PHASE
  After completing each phase, you MUST write a comprehensive markdown report
  inside a /reports folder at the root of the project. The report file must
  be named: phase-{N}-report.md (e.g., phase-1-report.md).
  See Section 5 for the exact required report structure.

RULE 4 — NEVER STORE MEDIA METADATA IN THE DATABASE
  Only IDs (tmdb_id, rawg_id) and user-generated data (status, rating,
  review, favorite, hours_played, episodes_watched) are stored in the
  database. ALL display data (title, poster, description, cast, genres)
  is fetched live from TMDB or RAWG APIs. Never deviate from this.

RULE 5 — ALL EXTERNAL API CALLS FROM BACKEND ONLY
  The frontend NEVER calls TMDB or RAWG directly. The frontend NEVER has
  access to TMDB_API_KEY or RAWG_API_KEY. All external API calls are made
  by the Express backend, which acts as a proxy. The frontend only calls
  your own backend API (VITE_API_URL).

RULE 6 — NEVER COMMIT SECRETS
  .env files are NEVER committed. Both /client/.env and /server/.env must
  be listed in their respective .gitignore files from the very first commit.
  .env.example files (with placeholder values, no real secrets) must be
  committed in their place.

RULE 7 — PARAMETERIZED QUERIES ONLY
  No SQL string interpolation. Ever. All database queries use the pg
  parameterized syntax: pool.query("SELECT ... WHERE id = $1", [id]).

RULE 8 — FOLLOW THE DOCUMENT
  The Software Architecture & Requirements Document defines the stack,
  schema, API routes, component names, folder structure, color system,
  typography, security measures, and deployment configuration. When in
  doubt about any implementation detail, the Document is the answer.
  Do not invent alternatives unless a documented approach is technically
  impossible, in which case you must explain why and propose a compliant
  alternative before proceeding.

RULE 9 — ALL FREE TIER
  Every service used in this project must be on its permanently free tier.
  No credit card charges. No paid upgrades. The full stack is:
    Frontend: Netlify (free)
    Backend:  Render (free)
    Database: Neon DB (free)
    Storage:  Cloudinary (free)
    APIs:     TMDB (free) + RAWG (free)

RULE 10 — WORKING CODE ONLY
  Do not write placeholder functions, TODO stubs, or commented-out logic
  and call the phase complete. Every function you write must do what its
  name says. Every endpoint must work. Every component must render.
  The phase is only complete when it functionally works end-to-end.

PHASE 2 — AUTHENTICATION SYSTEM
══════════════════════════════════════════════════════════════════════════════

GOAL: Complete, working auth system with JWT access tokens and refresh
tokens. A user can register, log in, refresh their session silently,
and log out.

  ⚠️  STOP POINT — JWT SECRET
  Before writing any auth code, STOP and tell the human:

  "I need you to generate a secure JWT secret and add it to your .env file.

  STEP 1: Run this command in your terminal:
          node -e "console.log(require('crypto').randomBytes(64).toString('hex'))"
          This generates a 128-character cryptographically secure random string.

  STEP 2: Copy the output and add it to /server/.env:
          JWT_SECRET=<paste here>

  STEP 3: Also add these values to /server/.env:
          JWT_EXPIRES_IN=24h
          REFRESH_TOKEN_EXPIRES_IN=7d

  STEP 4: Send me confirmation that these are set."

DELIVERABLES:

  [ ] /server/utils/AppError.js — custom error class with statusCode and code
  [ ] /server/validators/authValidator.js — validation rules:
        - register: username (3–50 chars, alphanumeric), email (valid format),
          password (min 8, 1 uppercase, 1 lowercase, 1 number)
        - login: email present + non-empty, password present + non-empty
        - passwordChange: oldPassword, newPassword (same policy as register)
  [ ] /server/models/userModel.js — functions:
        createUser(username, email, hashedPassword)
        findUserByEmail(email)
        findUserById(id)
        updateAvatar(userId, avatarUrl)
        updatePassword(userId, hashedPassword)
  [ ] /server/models/ — refresh token functions inside userModel or separate:
        createRefreshToken(userId, tokenHash, expiresAt)
        findRefreshToken(tokenHash)
        deleteRefreshToken(tokenHash)
        deleteAllRefreshTokensForUser(userId)
  [ ] /server/middleware/authMiddleware.js — verifies JWT, attaches req.user
  [ ] /server/middleware/rateLimiter.js — 5 attempts per 15 min on login
  [ ] /server/controllers/authController.js:
        register   — hash password, insert user, return 201
        login      — verify password, issue access token (JWT 24h) +
                     refresh token (opaque, hashed, stored in DB, 7d),
                     set refresh token in httpOnly cookie
        refresh    — validate refresh token cookie against DB hash,
                     issue new access token
        logout     — delete refresh token from DB, clear cookie
        getProfile — return current user (no password)
        changePassword — verify old password, hash new, update DB
  [ ] /server/routes/auth.js — wire all auth routes
  [ ] Mount auth routes in server.js: app.use('/api/auth', authRouter)

  FRONTEND AUTH:
  [ ] /client/src/context/AuthContext.jsx:
        - State: user, accessToken (in memory only), isLoading, isAuthenticated
        - Methods: login(), logout(), refreshToken(), updateUser()
        - On mount: attempt silent token refresh (calls /api/auth/refresh)
          to restore session on page reload
  [ ] /client/src/hooks/useAuth.js — consumes AuthContext
  [ ] /client/src/services/authService.js — login(), register(), logout(),
      refreshToken(), getProfile(), changePassword()
  [ ] Update /client/src/services/api.js:
        - Request interceptor: attach Authorization: Bearer <accessToken>
        - Response interceptor: on 401, call refreshToken(), retry once.
          If refresh fails, call logout()
  [ ] /client/src/routes/AppRouter.jsx — ProtectedRoute component that
      redirects to /login if not authenticated
  [ ] /client/src/pages/auth/Login.jsx — form with email + password,
      calls authService.login(), on success redirects to /dashboard
  [ ] /client/src/pages/auth/Register.jsx — form with username, email,
      password, calls authService.register(), on success redirects to /login
  [ ] /client/src/components/layout/Navbar.jsx — shows Login/Register
      buttons when logged out; shows avatar + dropdown (Profile, Settings,
      Logout) when logged in
  [ ] Wire all routes in AppRouter.jsx. Public routes: /, /login, /register.
      Protected routes: /dashboard, /library, /favorites, /lists, /reviews,
      /profile, /settings. Redirect authenticated users away from /login
      and /register.

DONE WHEN:
  - User can register with username/email/password
  - Duplicate email returns a clear error message
  - User can log in and receives a JWT
  - Refreshing the browser page keeps the user logged in (silent refresh)
  - Navigating to /dashboard while logged out redirects to /login
  - Logout clears the session completely
  - Login rate limiting works (6th attempt in 15 min returns 429)

-- TABLE 1: users
CREATE TABLE IF NOT EXISTS users (
  id               SERIAL PRIMARY KEY,
  username         VARCHAR(50)  UNIQUE NOT NULL,
  email            VARCHAR(255) UNIQUE NOT NULL,
  password         TEXT NOT NULL,
  avatar_url       TEXT DEFAULT '/images/default-avatar.png',
  email_verified   BOOLEAN DEFAULT FALSE,
  reset_token      TEXT,
  reset_token_expires TIMESTAMPTZ,
  created_at       TIMESTAMPTZ DEFAULT NOW()
);

-- TABLE 2: refresh_tokens
CREATE TABLE IF NOT EXISTS refresh_tokens (
  id         SERIAL PRIMARY KEY,
  user_id    INTEGER REFERENCES users(id) ON DELETE CASCADE,
  token_hash TEXT UNIQUE NOT NULL,
  expires_at TIMESTAMPTZ NOT NULL,
  created_at TIMESTAMPTZ DEFAULT NOW()
);

-- TABLE 3: tracked_movies
CREATE TABLE IF NOT EXISTS tracked_movies (
  id         SERIAL PRIMARY KEY,
  user_id    INTEGER REFERENCES users(id) ON DELETE CASCADE,
  tmdb_id    INTEGER NOT NULL,
  status     VARCHAR(20) CHECK (status IN ('watching','completed','planned','paused','dropped')),
  rating     SMALLINT CHECK (rating BETWEEN 1 AND 10),
  review     TEXT,
  favorite   BOOLEAN DEFAULT FALSE,
  watched_at TIMESTAMPTZ,
  created_at TIMESTAMPTZ DEFAULT NOW(),
  UNIQUE(user_id, tmdb_id)
);

-- TABLE 4: tracked_series
CREATE TABLE IF NOT EXISTS tracked_series (
  id               SERIAL PRIMARY KEY,
  user_id          INTEGER REFERENCES users(id) ON DELETE CASCADE,
  tmdb_id          INTEGER NOT NULL,
  status           VARCHAR(20) CHECK (status IN ('watching','completed','planned','paused','dropped')),
  rating           SMALLINT CHECK (rating BETWEEN 1 AND 10),
  review           TEXT,
  favorite         BOOLEAN DEFAULT FALSE,
  episodes_watched INTEGER DEFAULT 0,
  created_at       TIMESTAMPTZ DEFAULT NOW(),
  UNIQUE(user_id, tmdb_id)
);

-- TABLE 5: tracked_games
CREATE TABLE IF NOT EXISTS tracked_games (
  id           SERIAL PRIMARY KEY,
  user_id      INTEGER REFERENCES users(id) ON DELETE CASCADE,
  rawg_id      INTEGER NOT NULL,
  status       VARCHAR(20) CHECK (status IN ('playing','completed','backlog','paused','dropped')),
  rating       SMALLINT CHECK (rating BETWEEN 1 AND 10),
  review       TEXT,
  favorite     BOOLEAN DEFAULT FALSE,
  hours_played NUMERIC(6,1) DEFAULT 0,
  created_at   TIMESTAMPTZ DEFAULT NOW(),
  UNIQUE(user_id, rawg_id)
);

-- TABLE 6: custom_lists
CREATE TABLE IF NOT EXISTS custom_lists (
  id          SERIAL PRIMARY KEY,
  user_id     INTEGER REFERENCES users(id) ON DELETE CASCADE,
  name        VARCHAR(100) NOT NULL,
  description TEXT,
  created_at  TIMESTAMPTZ DEFAULT NOW()
);

-- TABLE 7: list_items
CREATE TABLE IF NOT EXISTS list_items (
  id         SERIAL PRIMARY KEY,
  list_id    INTEGER REFERENCES custom_lists(id) ON DELETE CASCADE,
  media_type VARCHAR(10) CHECK (media_type IN ('movie','series','game')),
  media_id   INTEGER NOT NULL,
  added_at   TIMESTAMPTZ DEFAULT NOW(),
  UNIQUE(list_id, media_type, media_id)
);

-- TABLE 8: activity_log
CREATE TABLE IF NOT EXISTS activity_log (
  id         SERIAL PRIMARY KEY,
  user_id    INTEGER REFERENCES users(id) ON DELETE CASCADE,
  action     VARCHAR(50) NOT NULL,
  media_type VARCHAR(10) NOT NULL,
  media_id   INTEGER NOT NULL,
  meta       JSONB,
  created_at TIMESTAMPTZ DEFAULT NOW()
);

-- INDEXES
CREATE INDEX IF NOT EXISTS idx_tracked_movies_user   ON tracked_movies(user_id);
CREATE INDEX IF NOT EXISTS idx_tracked_series_user   ON tracked_series(user_id);
CREATE INDEX IF NOT EXISTS idx_tracked_games_user    ON tracked_games(user_id);
CREATE INDEX IF NOT EXISTS idx_activity_log_user     ON activity_log(user_id, created_at DESC);
CREATE INDEX IF NOT EXISTS idx_refresh_tokens_user   ON refresh_tokens(user_id);
CREATE INDEX IF NOT EXISTS idx_list_items_list       ON list_items(list_id);
