## 🎯 1. Core Objectives

Neon Hub is a full-stack media tracking application designed to allow users to catalog, review, and discover Movies, TV Series, and Video Games in a single, unified interface.

## 📋 2. Functional Requirements (FR)

The system must provide the following capabilities to the users.

### User & Authentication Management

| ID | Requirement | Description |
|:---|:------------|:------------|
| **FR-01** | User Registration | Users must be able to create an account using a unique username, email, and password. |
| **FR-02** | User Authentication | Users must be able to log in securely to receive authentication tokens. |
| **FR-03** | Profile Management | Users must be able to view and edit their profiles, including uploading a custom avatar. |
| **FR-04** | Session Management | The system must automatically refresh user sessions transparently without logging them out. |
| **FR-05** | Guest Access | Unauthenticated users must be able to view trending media and search for content, but cannot add items to a library. |

### Media Discovery & Search

| ID | Requirement | Description |
|:---|:------------|:------------|
| **FR-06** | Cross-Media Search | The system must allow users to search for movies, series, and games simultaneously via a single search bar. |
| **FR-07** | Debounced Queries | Search inputs must be debounced to minimize unnecessary API calls. |
| **FR-08** | Detailed Metadata | Clicking a media item must display deep metadata (synopsis, cast, platforms, release date, genres, screenshots/backdrops). |
| **FR-09** | Trending Feeds | The homepage must display curated feeds of trending movies, series, and popular games. |

### Library & Tracking

| ID | Requirement | Description |
|:---|:------------|:------------|
| **FR-10** | Add to Library | Users must be able to add any media item to their personal library. |
| **FR-11** | Status Tracking | Users can track media by status (e.g., Planned, Watching, Completed, Dropped for Movies/Series; Backlog, Playing, Completed, Abandoned for Games). |
| **FR-12** | Ratings & Reviews | Users can rate media (1-10 scale) and attach text-based reviews. |
| **FR-13** | Favorite System | Users can mark any tracked media item as a "Favorite" for quick access. |
| **FR-14** | Hours Played | For video games, users can input and track the specific number of hours played. |

### Analytics & Organization

| ID | Requirement | Description |
|:---|:------------|:------------|
| **FR-15** | Dashboard Analytics | A personalized dashboard must display total tracked items, average rating, completion percentage, and a breakdown of top genres. |
| **FR-16** | Activity Feed | The dashboard must show a chronological feed of the user's latest interactions (e.g., "Added a review for Interstellar"). |
| **FR-17** | Custom Lists | Users must be able to organize their media into custom, named lists. |

### AI Integration

| ID | Requirement | Description |
|:---|:------------|:------------|
| **FR-18** | AI Recommendations | Users can request personalized media recommendations generated by AI based on their specific tracked library. |
| **FR-19** | AI Comparisons | Users can select exactly two media items and request an AI-generated side-by-side comparison of narrative, themes, and quality. |

## 🛡️ 3. Non-Functional Requirements (NFR)

The system must adhere to the following performance, security, and architectural constraints.

### Performance & Scalability

| ID | Requirement | Description |
|:---|:------------|:------------|
| **NFR-01** | Response Times | Core API endpoints must respond in under 200ms (excluding external 3rd-party API overhead). |
| **NFR-02** | External Caching | The server must cache frequent external requests (e.g., trending endpoints) in-memory for 10 minutes to avoid hitting rate limits. |
| **NFR-03** | AI Fallback | The system must gracefully fallback to alternate AI models if the primary model hits quota limits (HTTP 429). |
| **NFR-04** | Bundle Splitting | The React frontend must implement lazy loading to split code bundles, ensuring fast initial page loads. |

### Security & Privacy

| ID | Requirement | Description |
|:---|:------------|:------------|
| **NFR-05** | Secure Authentication | Authentication must utilize a two-token system: a short-lived, memory-only Access Token, and an HttpOnly, Secure Refresh Token cookie. |
| **NFR-06** | Password Hashing | All passwords must be hashed using `bcrypt` (minimum 12 rounds) before storage. |
| **NFR-07** | Rate Limiting | The login endpoint must restrict users to a maximum of 5 attempts per 15-minute window to prevent brute-force attacks. |
| **NFR-08** | Input Sanitization | All inputs and routes must be validated and sanitized to prevent SQL injection and Cross-Site Scripting (XSS). |
| **NFR-09** | HTTP Headers | The server must utilize Express Helmet to secure HTTP headers. |

### Usability & Accessibility

| ID | Requirement | Description |
|:---|:------------|:------------|
| **NFR-10** | Internationalization | The application must support both English (LTR) and Arabic (RTL) seamlessly. |
| **NFR-11** | Responsiveness | The UI must scale perfectly from mobile devices (320px width) up to 4K displays. |
| **NFR-12** | Theming | The application must utilize a consistent, premium dark-mode aesthetic. |
| **NFR-13** | SEO Optimization | Detail pages must dynamically generate `react-helmet` meta tags and Open Graph tags for rich link sharing. |

### Infrastructure

| ID | Requirement | Description |
|:---|:------------|:------------|
| **NFR-14** | Cloud Database | Data must persist in a normalized PostgreSQL database hosted on a serverless provider (Neon DB). |
| **NFR-15** | Media Storage | User avatars and user-generated images must be uploaded to and served from Cloudinary via a signed REST integration. |
